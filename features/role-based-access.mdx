## Overview

The Robotics School platform implements a robust role-based access control (RBAC) system with three distinct roles. Each role has specific permissions designed to support the educational mission while maintaining security and appropriate access levels.

## Three-Tier Role System

The platform uses an enum-based role system enforced at the database level, ensuring data integrity and preventing invalid role assignments.

```sql
$table->enum('rol', ['Student', 'Teacher', 'Administrative']);
```

## Role Permissions

### Student Role

**Access Level**: Read-only access to assigned content

<Accordion title="Student Permissions">
  **Can Do**:
  - View courses assigned to their group
  - Access course content and materials
  - View didactic materials and robotics kit information
  - See their group membership
  - Update their own profile information
  
  **Cannot Do**:
  - Create or edit courses
  - Manage other users
  - Change group assignments
  - Access administrative functions
  - View users outside their group
</Accordion>

### Teacher Role

**Access Level**: Full content management and student oversight

<Accordion title="Teacher Permissions">
  **Can Do**:
  - All student permissions
  - Create new courses
  - Edit and update course content
  - Upload cover images and didactic materials
  - Assign robotics kits to courses
  - Manage student groups and assignments
  - View student progress and engagement
  - Assign courses to groups
  
  **Cannot Do**:
  - Create or delete user accounts (depends on implementation)
  - Access system-wide administrative settings
  - Manage other teachers' permissions
  - Modify platform configuration
</Accordion>

### Administrative Role

**Access Level**: Full platform access and control

<Accordion title="Administrative Permissions">
  **Can Do**:
  - All teacher and student permissions
  - Create, edit, and delete user accounts
  - Assign roles to users
  - Manage all groups across the platform
  - Configure system-wide settings
  - Access analytics and reporting
  - Manage course library
  - Override any restrictions
  - Perform system maintenance tasks
  
  **Cannot Do**:
  - Nothing - administrative role has unrestricted access
</Accordion>

## Role Assignment

Roles are assigned when creating new user accounts:

1. Administrative users create new accounts
2. During creation, select one of the three available roles
3. The role is stored in the `rol` enum field
4. Permissions are automatically enforced based on the assigned role

<Note>
  Role assignments are permanent in the database structure but can be changed by administrative users through user management interfaces.
</Note>

## Security Features

### Password Security

- All passwords are hashed using Laravel's secure hashing functions
- Passwords are never stored in plain text
- The system supports strong password requirements

### Enum Validation

Roles are enforced at the database level using enum constraints:

```sql
$table->enum('rol', ['Student', 'Teacher', 'Administrative']);
```

This prevents:
- Invalid role values from being stored
- Typos or incorrect role names
- Unauthorized role escalation attempts

### Email Uniqueness

Each user must have a unique email address:

```sql
$table->string('email')->unique();
```

This ensures:
- No duplicate accounts
- Reliable user identification
- Secure password reset functionality

## Best Practices

<AccordionGroup>
  <Accordion title="Change Default Passwords Immediately">

  <Accordion title="Follow Principle of Least Privilege">
    Assign the minimum role necessary for each user's responsibilities. Don't grant administrative access unnecessarily.
  </Accordion>
  
  <Accordion title="Regular Permission Audits">
    Periodically review user roles and permissions to ensure they remain appropriate as responsibilities change.
  </Accordion>
  
  <Accordion title="Use Strong Passwords">
    Enforce strong password policies with minimum length, complexity requirements, and regular password rotation.
  </Accordion>
  
  <Accordion title="Monitor Administrative Actions">
    Log and review actions taken by administrative users to maintain accountability and security.
  </Accordion>
</AccordionGroup>

## Role-Based Workflows

### Student Workflow

1. Log in with student credentials
2. View assigned group and available courses
3. Access course content and materials
4. Complete learning activities with assigned robotics kits
5. Track personal progress

### Teacher Workflow

1. Log in with teacher credentials
2. Create or update course content
3. Upload didactic materials and specify robotics kits
4. Organize students into appropriate groups
5. Assign courses to groups
6. Monitor student engagement and progress

### Administrative Workflow

1. Log in with administrative credentials
2. Create teacher and student accounts
3. Configure groups and organizational structure
4. Oversee course library and content quality
5. Manage system-wide settings
6. Review platform analytics and usage
